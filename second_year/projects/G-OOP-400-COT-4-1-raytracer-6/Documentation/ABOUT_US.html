Architecture Globale

Le projet repose sur une architecture en couches strictement séparées. Chaque couche communique avec la suivante via des interfaces abstraites, garantissant un couplage faible et une extensibilité maximale.

Primitives Géométriques

Interface IPrimitive

Toutes les primitives implémentent cette interface pure. Elle définit le contrat minimal pour l'intersection rayon/surface.

C++ 
class IPrimitive {
public:
    virtual ~IPrimitive() = default;

    // Configure la primitive depuis un fichier de scène
    virtual void configure(const std::map<std::string, double>& params) = 0;

    // Retourne le point d'intersection ou nullopt
    virtual std::optional<HitRecord> intersect(
        const Ray& ray,
        double tMin = 1e-4,
        double tMax = std::numeric_limits<double>::infinity()
    ) const = 0;
};

Primitives disponibles

PrimitiveMéthode d'intersectionParticularités
Sphere Équation quadratique at² + bt + c = 0 Normale = (point − centre) / rayon
Plane Produit scalaire t = (C−O)·N / (D·N) Normale constante, rejet si rayon parallèle
Cylinder Projection sur plan XZ + bornes Y Hauteur finie, normale horizontale
Cone Quadratique avec pente k = r/h Apex en haut, hauteur limitée
Torus Équation quartique (méthode de Ferrari) Plus coûteux, sélection de la plus petite racine réelle positive
Triangle Algorithme de Möller-Trumbore Coordonnées barycentriques, backface culling

HitRecord — Données d'intersection

C++ 
struct HitRecord {
    double t;        // Distance sur le rayon
    Vec3   point;    // Point d'intersection dans le monde
    Vec3   normal;   // Normale à la surface
    bool   frontFace;// Face avant ou arrière
    Vec3   _color;   // Couleur de surface au point
};

TransformedPrimitive — Décorateur de transformation

Encapsule n'importe quelle IPrimitive pour lui appliquer une transformation sans modifier la primitive elle-même.

Pipeline de transformation

1. Le rayon est transformé en espace local (world → objet) · 2. L'intersection est calculée localement · 3. Le point et la normale sont retransformés en espace monde · 4. La normale utilise la matrice inverse-transposée pour préserver l'orthogonalité.

Transformations & Math

Vec3
Vecteur 3D
Opérateurs arithmétiques complets, produit scalaire (dot), produit vectoriel (cross), normalisation, reflection, réfraction, interpolation linéaire.
Mat4
Matrice 4×4
Stockage colonne-major, multiplication, transposée, inverse (élimination de Gauss avec pivotage partiel). Factories : identity, translation, rotationX/Y/Z, rotation arbitraire (Rodrigues), scale.

Matrices de transformation

C++ — Factories 
// Translation
Mat4 t = Mat4::translation(10.0, 0.0, 5.0);

// Rotations par axe
Mat4 rx = Mat4::rotationX(45.0);   // degrés → radians en interne
Mat4 ry = Mat4::rotationY(90.0);
Mat4 rz = Mat4::rotationZ(30.0);

// Rotation axe arbitraire (formule de Rodrigues)
Mat4 r  = Mat4::rotation(axis, angle);

// Transformation finale (TRS)
Mat4 transform = t * rx * ry * rz;

Transformation des normales

Les normales ne se transforment pas comme les points. Elles nécessitent la matrice inverse-transposée pour rester perpendiculaires à la surface sous des transformations non-uniformes :

Règle importante

Points : P' = M × P  ·  Normales : N' = (M⁻¹)ᵀ × N

Système d'Éclairage

Le module lights repose sur l'interface ILight et la classe Color. La fonction computeLighting() orchestre l'accumulation de toutes les sources pour un point donné.

Types de lumières

AmbientLight
Éclairage uniforme sans direction. Tous les points reçoivent la même contribution.

Calcul : surfaceColor × intensity
HitRecord ignoré (indépendant de la géométrie)
DirectionalLight
Source infiniment lointaine (soleil). Direction constante, pas d'atténuation.

Modèle Lambert :
lambert = max(0, normal · lightDir)
résultat = surfaceColor × diffuse × lambert
PointLight
Source ponctuelle avec atténuation quadratique physique.

att = 1 / (1 + 0.0001 × dist²)
résultat = surfaceColor × intensity × lambert × att

computeLighting() — Accumulation

C++ 
Color computeLighting(
    const HitRecord& hit,
    const Color& surfaceColor,
    const Scene& scene)
{
    Color result(0, 0, 0);

    for (auto& light : scene.findLights("ambient"))
        result += light->illuminate(hit, surfaceColor);
    for (auto& light : scene.findLights("directional"))
        result += light->illuminate(hit, surfaceColor);
    for (auto& light : scene.findLights("point"))
        result += light->illuminate(hit, surfaceColor);

    return result.clamped01();  // ← évite les débordements
}

Renderer & Sortie

CLI
Renderer
Rendu complet en une passe. Charge une scène .cfg, calcule tous les pixels, exporte via un IOutput (PPM par défaut). Résolution configurable via Dimensions.
GUI
SfmlOutput
Fenêtre SFML 1600×800 avec sélecteur de scènes, rendu progressif ligne par ligne, export PPM, états INTRO / MENU / RENDERER.

FrameBuffer

Conteneur 2D de couleurs. Accès en O(1) par coordonnées (x, y). Lève une RayTracerError si les coordonnées sont hors limites.

MéthodeDescription

IOutput — Interface de sortie

Abstraction de la destination du rendu. Toute nouvelle sortie (PNG, JPEG, réseau…) implémente simplement void render(const FrameBuffer&).

Caméra & Projection

Rectangle3D — Écran virtuel

Représente le plan de projection dans l'espace 3D. Défini par un coin origine et deux vecteurs directeurs.

C++ 
// Convertit (u, v) normalisés en point 3D sur l'écran
Vec3 convertTo3D(double u, double v) const {
    return _O + (_I * u) + (_J * v);
    // u, v ∈ [0.0, 1.0] → balaie tout l'écran
}

Camera::thorwRay()

Génère un rayon depuis la position caméra vers le pixel (u, v) de l'écran virtuel.

configCamera() — Algorithme

Étapes de configuration

1. Origin = position (xp, yp, zp)
2. FOV → halfW = tan(fov/2), halfH avec correction d'aspect ratio
3. Forward = cos(pitch)·sin(yaw), sin(pitch), cos(pitch)·cos(yaw) — normalisé
4. Cas dégénéré : si forward ∥ (0,1,0), worldUp = (0,0,1)
5. right = normalize(forward × worldUp) · up = normalize(right × forward)
6. screen = Rectangle3D(corner, right×2·halfW, up×2·halfH)

Système

Système de Plugins

Le moteur charge dynamiquement des primitives et lumières depuis des fichiers .so. Ajouter une nouvelle primitive ne nécessite pas de recompiler le moteur.

Export d'un plugin

C++ — Plugin export 
// extern "C" évite le name mangling du C++
extern "C" {
    IPrimitive* create() {
        return new Sphere();
    }
    const char* getType() {
        return "sphere";
    }
}

PluginLoader

Encapsule l'API POSIX : dlopen / dlsym / dlclose. Le loader reste vivant tant que la factory en a besoin — évite que les pointeurs de fonctions deviennent invalides.

PluginFactory<T>

Factory générique. La même logique gère PluginFactory<IPrimitive> et PluginFactory<ILight>. Les symboles create() sont résolus une seule fois à l'enregistrement.

C++ — Utilisation 
// Enregistrement automatique depuis .so
factory.loadPlugin("plugins/raytracer_sphere.so");

// Création à la demande, O(1)
auto sphere = factory.create("sphere");

SceneParser

Lit les fichiers .cfg via libconfig. Produit des structures de données pures (SphereData, PlaneData…) sans créer d'objets runtime. Supporte les transformations :

CFG — Exemple de scène 
spheres = (
{
    x = 0; y = 0; z = -5; r = 1;
    translation = { x = 2; y = 0; z = 0; };
    rotation    = { xr = 0; yr = 45; zr = 0; };
}
);

SceneBuilder

Prend les structures du parser, crée les primitives via la factory, construit la matrice de transformation composite et encapsule le tout dans un TransformedPrimitive avant de l'ajouter à la Scene.

Séparation des responsabilités

Le parser ne sait pas ce qu'est une primitive. Le builder ne sait pas comment parser. La factory ne connaît pas la scène. Chaque composant a un rôle unique.

Design Patterns

Factory
PluginFactory<T>, PrimitiveFactory
Création dynamique d'objets sans connaître leur type concret. Permet l'ajout de primitives sans modifier le moteur.
Builder
SceneBuilder
Construction progressive d'une scène complexe. Sépare le parsing du runtime et centralise la logique de création.
Decorator
TransformedPrimitive
Ajoute des transformations à n'importe quelle primitive sans la modifier. Composable et transparent pour le client.
Singleton
PrimitiveFactory::getInstance()
Instance unique de la factory. Thread-safe en C++11+ (Meyers' singleton), initialisation paresseuse.
Template Method
AbstractPrimitive
Définit la structure de l'algorithme dans la base. Les sous-classes implémentent les étapes spécifiques (calculateNormal).
RAII
unique_ptr, PluginLoader
Gestion automatique de la mémoire. Les plugins sont fermés (dlclose) dès que leur loader est détruit.

Pipeline de Rendu

Algorithme central exécuté pour chaque pixel. Identique en mode CLI (Renderer::render()) et GUI (SfmlOutput::showTheRendering()), à ceci près que le mode GUI rend une ligne par frame.

Pour chaque pixel (w, h) de l'image : │ ├── 1. LANCER DE RAYON │ u = w / (maxX - 1) ← coordonnée normalisée │ v = h / (maxY - 1) │ ray = camera.thorwRay(u, v) │ ├── 2. DÉTECTION D'INTERSECTION │ closest = nullopt │ Pour chaque primitive de la scène : │ hit = primitive.intersect(ray, tMin=1e-2, ∞) │ Si hit && hit.t < closestT → closest = hit │ ├── 3. CALCUL D'ÉCLAIRAGE │ Si closest existe : │ surfaceColor = Color(hit._color) │ lit = computeLighting(hit, surfaceColor, scene) │ ├── AmbientLight contributions │ ├── DirectionalLight (Lambert) │ └── PointLight (Lambert + atténuation) │ pixel = lit.clamped01() │ └── 4. ÉCRITURE PIXEL frameBuffer.setPixel(w, h, pixel * 255) Après tous les pixels : output.render(frameBuffer) → fichier .ppm / texture SFML
